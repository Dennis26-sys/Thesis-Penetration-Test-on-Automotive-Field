\chapter{Experimental Laboratory Setup}\label{ch:experimental_setup}

This chapter describes the experimental laboratory environment developed to analyse, simulate and test the \gls{CAN} network and the (\gls{UDS}) protocol that relies on \gls{CAN}. It describes the hardware testbed configuration, the design choices behind the software implementation and the physical and computational limitations encountered during penetration phases.

\section{Hardware Testbed Architecture}
\label{sec:hw_architecture}

The topology, prepared in controlled laboratory environment, to evaluate the security vulnerabilities is illustrated in Figure \ref{fig:Hardware_scheme}.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.95\textwidth]{images/Schema HW.png}
    \caption{Hardware architectural scheme of the simulated attacks tested in laboratory environment}
    \label{fig:Hardware_scheme}
\end{figure}

The core processing and implementation unit consists of a Raspberry Pi running a Kali Linux distribution. This platform serves as a high-privilege node inside the automotive architecture, directly connected to a physical \gls{CAN} bus via a dedicated transceiver shield interface. The network segment is in a reduced configuration, with the Raspberry Pi connected to a single physical \gls{ECU} simulating the server node. 

In this setup the Raspberry Pi provides a dual-role architectural purpose, as there is no full vehicle network topology:
\begin{itemize}
    \item \textbf{Adversary Node:} It performs the attacker's script to stress-test the target security properties that are shown in depth in Chapter \ref{ch:attacks}, primary targeting the lack of confidentiality and integrity in the simulated environment.
    \item \textbf{Network Virtualization Node:} The Raspberry Pi executes scripts to enable the \gls{ECU} to respond to injected packets, avoiding a power-saving mode. This background emulation reproduces the heartbeat frames, the regular cyclic operational data of vehicular network's \glspl{ECU} that are missing in the testbed while also providing the normal flow of packages transmitted during a diagnostic session.
\end{itemize}

\section{Software Environment}
\label{sec:sw_environment}

The software architecture deployed on the Kali Linux host is specifically designed to aid high-frequency bus monitoring, automated diagnostic identification and extensive penetration testing techniques. Rather than proprietary monolithic applications, the ecosystem employs the native Linux kernel network stack, customised automation layers, database engine integration and specialised offensive security instruments.

\subsection{Linux SocketCAN and Low-Level Core Libraries}
The architectural basis for the system is the Linux \textit{SocketCAN} subsystem, which interfaces the physical hardware transceiver as a regular network device (\texttt{can0}). This abstraction enables user-space applications to communicate with the vehicle bus via the common network socket \glspl{API}. The basic open-source libraries used to build the penetration framework are the following:
\begin{itemize}
    \item \textbf{\texttt{python-can}:} Provides object-oriented abstractions for the \gls{CAN} bus including the lifecycle management of a network interface listener, queues to safely transmit frames and state wrappers \cite{python_can_library}. In this setup it is heavily leveraged for its dynamic payload manipulation capabilities, allowing scripts to create the suitable payload for the specific necessities. In addition its native periodic broadcast scheduler (\texttt{bus.send\_periodic}) moves the timing-critical task of sending background traffic generation directly to the kernel or driver layer, thus ensuring consistent transmission intervals while not blocking the penetration script.
    \item \textbf{\texttt{queue}:} Used as a synchronization buffer for injection. A queue structure is used to keep track of the messages to be scheduled for transmission. This allows synthesised network traffic to be injected with tight timing characteristics, matching the original \gls{ECU} refresh rates and deceiving the network, making it difficult to recognise authentic from forged frames.
    \item \textbf{\texttt{isotp}:} Created to abstract the ISO 15765-2 transport protocol layer needed for high-level \gls{UDS} interactions \cite{python_isotp_library}. \gls{UDS} diagnostic sequences are request-response based. For this purpose the \texttt{isotp} module creates an ISO-\gls{TP} socket connection on top of the existing \texttt{python-can} interface. This allows for synchronous, linear read/write execution flows inside the Python scripts; the library automatically takes care of multi-frame packetization, flow control loops and multi-byte reassembly, in order to permit the program to block and wait for an incoming \gls{ECU} response before proceeding to the next diagnostic step.
    \item \textbf{\texttt{sqlite3} and \texttt{pandas}:} Used for big-data storage and execution metrics. Raw frames acquired during sniffing are streamed into a structured database, allowing analytical slicing and post-incident forensic analysis using relational queries and structural dataframes.
    \item \textbf{Asynchronous Primitives:} Utilizing Python's \texttt{threading.Event} controls, specialised background routines are implemented, such as a periodic \textit{Tester Present} heartbeat mechanism (\texttt{0x3E}) to keep a diagnostic session open while scripts run tasks downstream.
\end{itemize}

\subsection{Automotive Security Framework: Caring Caribou}
\textbf{Caring Caribou} is an industry-standard open-source penetration testing framework, customised for automotive \glspl{ECU} and a core component of the offensive software toolkit \cite{caring_caribou_github}. It is built into the execution pipeline through Python's \texttt{subprocess} \gls{API} to automate exploratory phases that would otherwise be computationally prohibitive if implemented manually from scratch. 

In this laboratory setup, two major capabilities of the framework are extensively leveraged:

\subsubsection*{UDS Discovery}
The framework executes the automated discovery primitive to ascertain diagnostic capabilities in the connected server node:

\begin{lstlisting}[language=bash, caption={Caring Caribou UDS Discovery Command Structure}]
caringcaribou -i <interface> uds discovery -min <min_value> -max <max_value>
\end{lstlisting}

This module scans a defined range of Arbitration Identifiers to detect listening \glspl{ECU}. It systematically finds the diagnostic address by listening to the positive response and gather the findings on screen.

\subsubsection*{Automated Security Seed Capturing}
Caring Caribou automates the interaction with the \textit{Security Access} (\texttt{0x27}) service, allowing cryptographic and entropy evaluation, using the following command:

\begin{lstlisting}[language=bash, caption={Caring Caribou Security Seed Capture Command Syntax}]
caringcaribou -i can0 uds security_seed -r 3 -n 200 0x02 0x01 0xAAAAAAAA 0xBBBBBBBB
\end{lstlisting}

This particular implementation deals with the full diagnostic routine lifecycle, automatically resetting the \gls{ECU} session state (using parameter \texttt{-r 3}), requesting a fixed number of seeds (parameter \texttt{-n 200}) for a specific diagnostic session and security level; handling the explicit physical addressing routing from source transmitter identifier (\texttt{transmitter id: 0xAAAAAAAA}) to the corresponding destination receiver identifier (\texttt{receiver id: 0xBBBBBBBB}). The resulting seed distribution data is then stored into a database to identify entropy degradation or \gls{PRNG} vulnerabilities.

\subsection{Database Architecture and Data Persistence Layer}
To support both real-time analytical monitoring and long-term security evaluations, the software environment isolation layer integrates two distinct database engines. These local storage solutions are decoupled to meet two different functional requirements within the testbed: telemetry logging and stateful cryptographic accounting.

\subsubsection*{Network Traffic Logging Database}

The first database operates as a high-performance network sniffer, continuously archiving raw traffic generated across the vehicular bus interface. \gls{CAN} communications are very frequent, with hundreds of messages being broadcast in milliseconds, so this storage engine is optimised for fast inserts. The underlying table schema is designed with four key columns aimed at reconstructing the network state during attack path evaluation or post-incident forensic analysis:
\begin{itemize}
    \item \textbf{Arbitration ID:} Contains the \gls{CAN} identifier, which enables a filtering of messages. This column is used in fuzzing attack to understand which input can affect the internal state of the vehicle and understand the topology of the vehicle.
    \item \textbf{Payload:} The raw data  bytes transported by the frame in a binary or hexadecimal string. This field is useful to analyse to check for counter and checksum field that can be inferred from a collection of frames.
    \item \textbf{Interface:} Log the specific network interface socket (e.g., \texttt{can0}) to enable multi-bus mapping capabilities in case of multiple transceivers deployment.
    \item \textbf{Timestamp:} A high-resolution time primitive that records the exact epoch at which the frame was received by the Linux kernel socket, which allows for accurate inter-frame jitter and cyclic frequency calculations.
\end{itemize}

\subsubsection*{Cryptographic Security Access Database}
The second database is focused on the analysis of the entropy, predictability and robustness of the diagnostic security layer. This engine differs from the network logger model in the fact that it is a stateful ledger only for tracking the \textit{Security Access} (\texttt{0x27}) service request loops. Its schema is carefully designed to show statistical anomalies in the \gls{ECU}'s \gls{PRNG} by grouping collected diagnostic challenges into three columns:
\begin{itemize}
    \item \textbf{Seed:} Stores the unique challenge sent by the server ECU after a diagnostic authorization request.
    \item \textbf{Key:} Records the corresponding cryptographic response calculated or injected to unlock the security level.
    \item \textbf{Counter:} An atomic integer that counts the number of occurrences for each seed value. If an active script or testing framework captures a seed that already exists in the ledger, the database will increment this field rather than creating a duplicate row.
\end{itemize}
This aggregation logic lays the mathematical foundation required to compute seed distribution patterns, identify recurrence vulnerabilities and detect static or poorly randomised cryptographic challenges within the connected \gls{ECU} firmware.

\section{Main Limitations}
\label{sec:main_limitations}

While the experimental setup provides a highly flexible and isolated environment for protocol evaluation, some hardware and architecture constraints need to be formalised to bound the validity of the empirical results:

\begin{itemize}
    \item \textbf{Non-Real-Time Operating System Scheduling:} the Raspberry Pi is running a general-purpose Linux kernel as opposed to industrial automotive testing equipment or bare metal. The environment is optimized for penetration testing and packet manipulation but the scheduler is not guaranteed to  have strict real-time deterministic behaviour. Therefore execution time, context-switching overheads and non-deterministic process preemption can cause jitter in providing rapid sequential frames.
    \item \textbf{Memory Buffering and Computational Constraints:} The volatile memory (\gls{RAM}) capacity and bus bandwidth of the platform impose rigid upper bounds to high-frequency packet capture operations. Continuous sniffing while performing memory demanding operations like bruteforcing can lead to an exhaustion of resources and must be accounted for when developing the code for penetration testing.
    \item \textbf{Hardware and Driver-Level Structural Constraints:} Constraints on the software injection engine are derived from the architectural separation between the user-space applications and the low-level network driver stack (\textit{SocketCAN}). User scripts can change timing intervals and the application payload but not structural frame parameters such as the Data Length Code (\gls{DLC}) or transport-layer state control flags. The underlying hardware controller and kernel-level drivers automatically validate and overwrite these fields before scheduling them onto the physical bus, thus preventing injection of structurally non-compliant or corrupted protocol headers to exploit parser vulnerabilities in the target \gls{ECU} without direct modification of the firmware of the network interface.
   \item \textbf{Isolated Node Topology and Lack of Error Propagation Mapping:} The physical testbed is based on a minimised network architecture, consisting of only one physical \gls{ECU}. Although this deployment enables the validation of targeted client-server transactions, the lack of a full multi-node topology precludes the experimental analysis of horizontal scaling of exploits. For example, cross-domain error propagation, cascading bus-offs, or downstream behavioural anomalies that malicious payloads could cause within independent gateway-bridged automotive subsystems cannot be evaluated. 
\end{itemize}
